# PLAT-0002: Интеграция онтологии АИС в SpecKit-фреймворк

## Контекст и постановка проблемы

SpecKit управляет спецификациями через machine-readable артефакты (requirements.yml, OpenAPI, AsyncAPI, SLO) с CI-валидацией. При этом фреймворк не имеет структурированного словаря архитектурных понятий уровня ИС (информационной системы): нет трёхслойной модели архитектуры, нет типовых архитектурных представлений, нет классификационных кодов компонентов и нет привязки NFR к отраслевым стандартам качества ПО.

Для команд, разрабатывающих системы класса АИС (по методологии ГосТех / ArchiMate), требуется соответствующая архитектурная основа внутри SpecKit.

Источник: «Онтология и методологические рекомендации по использованию при проектировании архитектуры АИС «Налог-3»», ГНИВЦ / ФНС, 2025. Основана на ArchiMate 3.2 (спецификация Open Group, 2022), ГОСТ Р 57100-2016, ГОСТ Р ИСО/МЭК 25020-2023.

## Факторы принятия решения

- **Единая терминология**: разные команды должны использовать одни и те же определения архитектурных понятий
- **Machine-readable first**: новые артефакты должны поддаваться CI-валидации, как существующие
- **Backwards compatibility**: добавление не должно ломать существующие requirements.yml и CI gates
- **Постепенное освоение**: интеграцию следует вводить поэтапно, начиная с глоссария

## Рассмотренные варианты

- **Вариант A:** Не делать ничего — каждая команда самостоятельно описывает архитектуру
- **Вариант B:** Добавить ссылку на PDF-документ в README — пассивная ссылка без структурирования
- **Вариант C:** Полная интеграция онтологии в SpecKit — L1-домен, расширение шаблонов, JSON Schema, новые команды

## Решение

**Выбранный вариант: C — поэтапная полная интеграция.**

Обоснование: Варианты A и B не обеспечивают ни единой терминологии, ни machine-readable артефактов, ни возможности CI-валидации. Вариант C добавляет ценность на каждом этапе и может быть остановлен после любого этапа при необходимости.

### Детали реализации (шесть этапов)

**Этап 1 (приоритетный):** Создать L1-домен `domains/is-ontology/` с:
- `glossary.md` — все ~34 термина из раздела 3 онтологии (Система, ИС, АИС, Подсистема, Сервис, Микросервис, Платформа, ...)
- `canonical-model/model.md` — трёхслойная модель (слой деятельности / приложений / технологический) с таблицами элементов каждого слоя
- `relationship-catalog/relationships.md` — таксономия 9 типов архитектурных взаимосвязей
- `nfr/domain-nfr.md` — характеристики качества по ГОСТ Р ИСО/МЭК 25020-2023

**Этап 2:** Расширить шаблон `initiatives/{INIT}/design.md` секциями «Architecture Layers» и «Architecture Views» (11 типов представлений)

**Этап 3:** Расширить `tools/schemas/requirements.schema.json`: classification-поля (system_scale, subsystem_type) и iso_characteristic в NFR-метриках — все новые поля опциональные

**Этап 4:** Добавить профиль «Enterprise IS» в `.specify/memory/constitution.md`

**Этап 5:** Создать команду `/speckit-architecture` для Claude Code

**Этап 6:** Демо-инициатива как валидация подхода

### Последствия

- **Хорошо:** Единая терминология архитектурных понятий для всех команд
- **Хорошо:** Возможность использовать SpecKit на ИС-проектах АИС-класса
- **Хорошо:** NFR в requirements.yml приобретают привязку к ГОСТ 25020
- **Нейтрально:** Новые артефакты на русском языке (онтология написана по-русски)
- **Плохо:** Увеличение объёма шаблонов — команды, не работающие с ИС, видят избыточные секции

### Подтверждение

Решение считается подтверждённым, когда:
1. `make validate` проходит без ошибок на existing requirements.yml (backwards compat)
2. `make validate` проходит на demo-инициативе с новыми classification полями
3. Глоссарий охватывает все ~34 определения из раздела 3 онтологии
