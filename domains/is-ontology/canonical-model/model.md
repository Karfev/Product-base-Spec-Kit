**Домен:** is-ontology
**Версия:** 1.0.0
**Последнее обновление:** 2026-03-03
**Источник:** Онтология АИС «Налог-3», ГНИВЦ / ФНС, 2025. ArchiMate 3.2, ГОСТ Р 57100-2016.

## 1. Общая структура трёхслойной модели

Архитектура ИС описывается в трёх слоях. Каждый слой предназначен для определённой аудитории и уровня детализации.

```
┌──────────────────────────────────────────────────────────────────┐
│  СЛОЙ ДЕЯТЕЛЬНОСТИ  (цвет: жёлтый)                               │
│  Для кого: бизнес-аналитики, руководители, заказчики             │
│  Что описывает: процессы, участники, события, объекты бизнеса    │
├──────────────────────────────────────────────────────────────────┤
│  СЛОЙ ПРИЛОЖЕНИЙ  (цвет: бирюзовый)                              │
│  Для кого: архитекторы ПО, разработчики, системные аналитики     │
│  Что описывает: компоненты ПО, сервисы, объекты данных           │
├──────────────────────────────────────────────────────────────────┤
│  ТЕХНОЛОГИЧЕСКИЙ СЛОЙ  (цвет: зелёный)                           │
│  Для кого: DevOps-инженеры, администраторы инфраструктуры        │
│  Что описывает: узлы, системное ПО, артефакты развёртывания      │
└──────────────────────────────────────────────────────────────────┘
```

Связь слоёв: элементы нижнего слоя **реализуют** элементы верхнего слоя.

## 2. Таксономия элементов

### 2.1 Общие типы элементов (применяются на всех слоях)

Элементы подразделяются на три базовых типа в зависимости от роли в модели:

| Базовый тип | Углы | Цвет | Назначение |
|-------------|------|------|------------|
| **Активный структурный** | Прямые | Тёмно-серый | Субъект, способный выполнять действия |
| **Элемент поведения** | Скруглённые | Серый | Поведение, выполняемое активными элементами |
| **Пассивный структурный** | Прямые | Светло-серый | Объект, над которым выполняется действие |

Дополнительно:

| Базовый тип | Назначение |
|-------------|------------|
| **Составной элемент** | Контейнер для группировки других элементов |

### 2.2 Слой деятельности (Business Layer)

| Элемент | Базовый тип | Подтип | Определение |
|---------|-------------|--------|-------------|
| Участник деятельности | Активный структурный | Внутренний | Субъект (роль, команда, организация), выполняющий функции в рамках деятельности |
| Интерфейс деятельности | Активный структурный | Внешний | Точка доступа, через которую участники взаимодействуют с системой |
| Процесс деятельности | Поведение | Внутренний | Последовательность поведений, позволяющих достичь определённого результата |
| Функция | Поведение | Внутренний | Последовательность действий, выполняемых определённой ролью, формирующая определённый результат |
| Событие деятельности | Поведение | Внешний | Состояние, которое влияет или контролирует дальнейшее выполнение процесса |
| Объект деятельности | Пассивный структурный | — | Объект (информация, документ, продукт), с которым взаимодействуют процессы |
| Система/Подсистема | Составной | — | Элемент-контейнер, объединяющий другие элементы слоя деятельности |

### 2.3 Слой приложений (Application Layer)

| Элемент | Базовый тип | Подтип | Определение |
|---------|-------------|--------|-------------|
| Компонент приложения | Активный структурный | Внутренний | Инкапсулированная часть функциональности приложения |
| Комплекс приложений | Активный структурный | Внутренний | Совокупность двух или более компонентов приложения, работающих вместе |
| Интерфейс приложения | Активный структурный | Внешний | Точка доступа, через которую другие компоненты получают доступ к функциональности |
| Объект данных | Пассивный структурный | — | Данные, которыми управляет приложение (структуры, схемы, модели) |

### 2.4 Технологический слой (Technology Layer)

| Элемент | Базовый тип | Подтип | Определение |
|---------|-------------|--------|-------------|
| Узел | Активный структурный | Внутренний | Вычислительный ресурс (сервер, ВМ, контейнер), на котором размещаются артефакты |
| Системное ПО | Активный структурный | Внутренний | Программное обеспечение, управляющее аппаратным оборудованием (ОС, СУБД, брокеры) |
| Артефакт | Пассивный структурный | — | Физическое проявление ПО: исполняемый файл, Docker-образ, JAR, конфигурация |

## 3. Типы архитектурных представлений (11 видов)

Каждый тип представления отвечает на конкретные вопросы.

### 3.1 Слой деятельности — 6 представлений

| № | Наименование | Основные вопросы |
|---|--------------|-----------------|
| Д-1 | Схема внешнего взаимодействия деятельности | Кто внешние участники? Какие интерфейсы предоставляются? Как взаимодействуют внешние системы? |
| Д-2 | Схема внутреннего взаимодействия деятельности | Как взаимодействуют внутренние подсистемы? Какие интерфейсы используются между ними? |
| Д-3 | Схема внешних потоков данных деятельности | Какие данные поступают извне и уходят наружу? Каков формат и направление потоков? |
| Д-4а | Схема процесса деятельности — 1-й уровень (Контекст) | Что является входом и выходом процесса? Кто участник? Каков контекст? |
| Д-4б | Схема процесса деятельности — 2-й уровень (Состав) | Из каких шагов состоит процесс? Кто выполняет каждый шаг? |
| Д-5 | Схема внутренних потоков данных | Как данные циркулируют внутри системы/подсистемы? |
| Д-6 | Функциональная карта | Какие функции реализует система? Как они структурированы по подсистемам? |

### 3.2 Слой приложений — 2 представления

| № | Наименование | Основные вопросы |
|---|--------------|-----------------|

| П-2 | Схема внутреннего взаимодействия компонентов | Из каких компонентов состоит подсистема? Как они взаимодействуют между собой? |

## 4. Классификационная система

### 4.1 Классификация систем

**По масштабу (вид: С.М)**

| Код | Масштаб | Критерий |
|-----|---------|----------|
| С.М.М | Малый | < 10 млн строк кода |
| С.М.С | Средний | ≥ 10 млн и < 100 млн строк кода |
| С.М.Б | Большой | ≥ 100 млн строк кода |

### 4.2 Классификация подсистем

**По типу (вид: ПС.Т)** — классы взаимоисключающи:

| Код | Тип | Описание |
|-----|-----|----------|
| ПС.Т.И | Инфраструктурная | ПО, создающее вычислительную инфраструктуру |
| ПС.Т.ПТ | Программно-технологическая | ПО для создания и управления жизненным циклом ПО |
| ПС.Т.П | Прикладная | ПО, решающее задачи конкретной предметной области |

**По масштабу (вид: ПС.М)**:

| Код | Масштаб |
|-----|---------|
| ПС.М.М | Малый |
| ПС.М.С | Средний |
| ПС.М.Б | Большой |

**По виду деятельности (вид: ПС.В)** — выбирается из справочника видов деятельности организации.

**По владельцу (вид: ПС.ВЛ)** — организация, подразделение или роль, ответственные за подсистему.

### 4.2.1 Руководство по выбору кодов (Decision Guide)

**Как выбрать `system_scale`:**

| Вопрос | Ответ | Код |
|--------|-------|-----|
| Каков суммарный объём кода системы? | < 10 млн строк | `С.М.М` |
| Каков суммарный объём кода системы? | 10–100 млн строк | `С.М.С` |
| Каков суммарный объём кода системы? | > 100 млн строк | `С.М.Б` |
| Объём кода неизвестен — как оценить? | < 10 сервисов по < 50k LOC каждый | `С.М.М` |
| Объём кода неизвестен — как оценить? | 10–100 сервисов или общий объём > 500k LOC | `С.М.С` |

**Как выбрать `subsystem_type`:**

| Вопрос | Если да | Код |
|--------|---------|-----|
| Создаёт ли подсистема вычислительную инфраструктуру (сети, хранилища, оркестрация)? | — | `ПС.Т.И` |
| Является ли подсистема инструментом разработки, CI/CD, платформой? | — | `ПС.Т.ПТ` |
| Решает ли подсистема задачи конкретной предметной области (бизнес-логика)? | — | `ПС.Т.П` |

> Если подсистема одновременно решает бизнес-задачи И предоставляет платформенные возможности — выбирайте по **основной** функции (той, ради которой создана подсистема).

### 4.3 Классификация ПО (9 видов)

| Вид | Примеры классов |
|-----|----------------|
| По функциональному назначению | ПО для управления данными, ПО интеграции, ПО мониторинга... |
| По типу | Программный компонент, программный комплекс |
| По варианту применения | Коробочный продукт, заказная разработка, open source |
| По степени автоматизации | Автоматизированный, автоматический |
| По способу приобретения | Собственная разработка, покупка, аренда (SaaS) |
| По уровню размещения | На узле, в контейнере, в облаке |
| По уровню сетевой зависимости | Онлайн, офлайн, гибридный |
| По типу пользователя | Для конечных пользователей, для разработчиков, для систем |
| По способам взаимодействия | Синхронный (REST/gRPC), асинхронный (очереди), пакетный |

### 4.4 Классификация сервисов ПО

**По типу взаимодействия:**

| Тип | Описание |
|-----|----------|
| Синхронный | Запрос–ответ в рамках одного цикла (REST, gRPC, GraphQL) |
| Асинхронный | Сообщения через брокер (Kafka, RabbitMQ, AMQP) |
| Пакетный | Периодическая обработка данных без ожидания ответа |

## 5. Формирование наименований

### 5.1 Наименование подсистемы

Формат: `[Класс функционального назначения] «[Функциональное наименование]» [Системы]`

Пример: Прикладная подсистема «Управление API-ключами» АИС «Налог-3»

### 5.2 Наименование ПО

Формат: `[Аббревиатура] [Основной класс функционального назначения и тип] «[Функциональное наименование]»`

Пример: ПКТ МПУПД «Мой налог» — Прикладной компонент «Мобильное приложение для учёта профессионального дохода» «Мой налог»

**Правила:**
- Кодовое наименование допустимо только если функциональное сложно для запоминания
- Кодовое наименование: не более двух слов
- Все наименования — в именительном падеже

## 6. Архитектурные принципы

### 6.1 Функциональные принципы

| Принцип | Суть |
|---------|------|
| Клиентоцентричное проектирование | Проектирование от потребностей пользователя, а не от технических возможностей системы |
| Омниканальный подход | Единый пользовательский опыт через все каналы взаимодействия |
| Унификация пользовательского интерфейса | Единые стандарты UI/UX для всех компонентов системы |

### 6.2 Архитектурные принципы

| Принцип | Суть |
|---------|------|
| Использование сервисов облачной платформы | Предпочтение платформенным сервисам перед собственной инфраструктурой |
| Сервис-ориентированная архитектура (SOA) | Функциональность реализуется как независимые сервисы с чётко определёнными интерфейсами |
| Единая техническая политика | Соблюдение утверждённых стандартов и шаблонов при проектировании |

### 6.3 Организационные принципы

| Принцип | Суть |
|---------|------|
| Единая архитектурная политика | Единые правила и стандарты для всех команд, участвующих в разработке |
| Производственный конвейер | CI/CD-процессы обеспечивают качество и скорость поставки |
| Экономическая эффективность | Архитектурные решения принимаются с учётом стоимости реализации и владения |
