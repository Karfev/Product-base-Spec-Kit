# PRD: User Notification Preferences

**Initiative:** INIT-2026-002-notification-preferences
**Owner (PM):** @platform-team
**Last updated:** 2026-04-10
**Profile:** Standard

---

## Цель и ожидаемый эффект

- **Проблема:** Пользователи не могут управлять тем, какие уведомления получают, по каким каналам и с какой частотой. Все уведомления включены по умолчанию, что приводит к отпискам и росту обращений в поддержку.
- **Почему сейчас:** GDPR-аудит выявил отсутствие гранулярного opt-out. Объём тикетов поддержки по теме "отключить уведомления" растёт на 20%/месяц.
- **Цель (Outcome):** Пользователь может самостоятельно настроить предпочтения уведомлений, что снизит нагрузку на поддержку и обеспечит GDPR compliance.

## Пользователи и сценарии

- **Primary personas:**
  - End User — хочет тишины, настраивает каналы и частоту
  - Product Admin — хочет видеть метрики opt-out и delivery

- **Top JTBD / сценарии:**
  1. Пользователь открывает настройки и отключает маркетинговые уведомления (REQ-NOTIF-004)
  2. Пользователь переключает канал доставки с email на push (REQ-NOTIF-002)
  3. Пользователь выбирает еженедельный дайджест вместо мгновенных уведомлений (REQ-NOTIF-003)

## Метрики успеха

| Метрика | Baseline | Target | Период | Источник |
|---|---:|---:|---|---|
| Тикеты поддержки "уведомления" | 120/мес | 60/мес | 60d | Zendesk |
| Opt-out rate | 12% | <5% | 90d | Analytics |
| API latency P95 | — | <200ms | 30d | APM |

## Scope

**In-scope:**
- Получение текущих предпочтений (REQ-NOTIF-001)
- Управление каналами доставки: email, push, SMS (REQ-NOTIF-002)
- Настройка частоты: immediate / daily / weekly digest (REQ-NOTIF-003)
- Opt-out по категориям уведомлений (REQ-NOTIF-004)
- API latency SLO (REQ-NOTIF-005)
- Событие при изменении предпочтений (REQ-NOTIF-006)

**Non-goals:**
- Управление контентом/шаблонами уведомлений
- Инфраструктура доставки (push-провайдер, SMTP)
- Admin UI для управления категориями

## Риски и ограничения

- **GDPR compliance:** Mandatory-категории (security) нельзя отключать — нужна чёткая бизнес-логика. Mitigation: is_mandatory флаг на категории.
- **Обратная совместимость:** Существующие пользователи не имеют preferences — нужны дефолты. Mitigation: lazy initialization при первом запросе.
- **Нагрузка:** Массовое изменение preferences при launch может вызвать пиковую нагрузку. Mitigation: rate limiting на API.
