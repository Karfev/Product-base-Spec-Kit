# PRD: API Key Management

**Initiative:** INIT-2026-000-api-key-management
**Owner (PM):** @platform-team
**Last updated:** 2026-03-01
**Profile:** Standard

---

## Цель и ожидаемый эффект

- **Проблема:** Клиенты B2B используют сервисные аккаунты для интеграций, которые требуют
  долгосрочной аутентификации. Bearer token-сессии не подходят из-за истечения срока действия.
  Отсутствие API-ключей блокирует enterprise-интеграции.
- **Почему сейчас:** 3 enterprise-клиента заблокированы на этапе интеграции. Потенциальный MRR под риском: $45k. Квартальный дедлайн — Q1 2026.
- **Цель (Outcome):** Предоставить командам возможность создавать API-ключи для программного
  доступа к платформе без использования логина/пароля. NPS enterprise-интеграций +15 пунктов
  в течение квартала.

## Пользователи и сценарии

- **Primary personas:** Platform integrators (backend-разработчики на стороне клиента)
- **Top JTBD / сценарии:**
  1. Разработчик создаёт API-ключ в консоли и использует его в CI/CD pipeline для автоматизированного доступа к API платформы
  2. Администратор аккаунта отзывает скомпрометированный API-ключ, не затрагивая другие ключи
  3. Разработчик выставляет срок действия ключа для временного доступа подрядчика

## Scope

**In-scope:**

- Создание API-ключей с именем и опциональным сроком действия (REQ-AUTH-001, REQ-AUTH-005)
- Отзыв API-ключей с распространением в течение 60 секунд (REQ-AUTH-002)
- Листинг API-ключей без раскрытия секрета (REQ-AUTH-003)
- Аутентификация запросов с P95-задержкой < 10ms (REQ-AUTH-004)

**Non-goals:**

- UI-консоль управления ключами (Phase 2)
- Автоматическая ротация ключей
- Scope-ограничения / fine-grained permissions для ключей
- Webhook-уведомления о приближающемся истечении срока

## Риски и ограничения

- **Компрометация секрета:** Секрет возвращается только один раз при создании. Если утерян — пользователь должен создать новый ключ. Mitigation: явная документация этого поведения в UI и API.
- **Задержка отзыва:** Redis-кэш TTL 60s создаёт окно компрометации. Mitigation: явная инвалидация кэша при DELETE, TTL только как fallback.
- **Производительность bcrypt:** Work factor 10 = ~100ms хэширования. Mitigation: хэширование только при создании, не при каждом запросе; auth проверяет только хэш.

## Требования (ссылки на REQ)

Реестр требований — в `requirements.yml`. Здесь только ссылки:

- `REQ-AUTH-001` (P0): Создание API-ключа — аутентифицированный пользователь создаёт именованный ключ, секрет возвращается один раз
- `REQ-AUTH-002` (P0): Отзыв API-ключа — немедленная инвалидация, распространение < 60s
- `REQ-AUTH-003` (P1): Листинг API-ключей — без секрета в ответе
- `REQ-AUTH-004` (P1): SLO задержки аутентификации — P95 < 10ms
- `REQ-AUTH-005` (P1): Опциональное истечение срока действия ключа — HTTP 401 KEY_EXPIRED
