## Твоя последовательность действий

### Шаг 0 — Прочитать контекст

1. Прочитай `initiatives/$ARGUMENTS/prd.md` (если есть) — понять цель и акторов
2. Прочитай `initiatives/$ARGUMENTS/design.md` — посмотреть, что уже заполнено
3. Прочитай `domains/is-ontology/canonical-model/model.md` — освежить типы элементов
4. Если design.md содержит незаполненные `{placeholder}`, продолжай заполнять их
5. Проверь поле `metadata.profile` в `initiatives/$ARGUMENTS/requirements.yml`

> **Ветвление по профилю:**
> - `enterprise` — выполни все шаги 1–6 включая классификацию и полные таблицы слоёв
> - `standard` / `extended` / `minimal` — выполни шаги 1–3 и 5–6 (шаг 4 пропусти; таблицы элементов не генерируй; вопросы об архитектурных представлениях предложи по умолчанию: Д-1, Д-3, П-1, Т-1)

### Шаг 1 — Слой деятельности (жёлтый)

Задай архитектору **последовательно** следующие вопросы. Жди ответа перед следующим вопросом.

```
1. Кто является внешними участниками (акторами) этой подсистемы?
   (примеры: Platform Integrator, End User, Admin, External System)

2. Через какие интерфейсы взаимодействуют участники с подсистемой?
   (примеры: REST API, веб-портал, очередь сообщений, файловый обмен)

3. Какие бизнес-процессы реализует подсистема?
   (примеры: создание API-ключа, аутентификация запроса, отзыв доступа)

4. Какие функции выполняет подсистема (операции конкретной роли)?
   (примеры: валидация API-ключа, генерация секрета)

5. Какие бизнес-объекты (данные, документы) обрабатывает подсистема?
   (примеры: API-ключ с полями id/name/expires_at, Токен доступа)
```

### Шаг 2 — Слой приложений (бирюзовый)

```
6. Какие программные компоненты реализуют функции подсистемы?
   (примеры: KeyService, AuthMiddleware, NotificationService)

7. Какие интерфейсы приложений предоставляет подсистема?
   (примеры: REST endpoints POST/GET/DELETE, gRPC-интерфейс, топик Kafka)

8. Какие объекты данных/схемы используются?
   (примеры: ApiKey schema, AccessToken schema — ссылки на contracts/schemas/*.json)
```

### Шаг 3 — Технологический слой (зелёный)

```
9. Какие узлы развёртывания используются?
   (примеры: App Server (горизонт. масштаб), PostgreSQL, Redis Cluster, S3)

10. Какое системное ПО работает на узлах?
    (примеры: PostgreSQL 15, Redis 7, Node.js 20, Kubernetes 1.29)

11. Какие развёртываемые артефакты создаются?
    (примеры: Docker-образ api-keys-service:latest, migration_001.sql, Helm chart)
```

### Шаг 4 — Классификация подсистемы _(только для Enterprise; для других профилей пропусти)_

```
12. Масштаб системы (выбери одно):
    С.М.М — малая (<10M LOC строк кода)
    С.М.С — средняя (10–100M LOC)
    С.М.Б — большая (>100M LOC)

13. Тип подсистемы (выбери одно):
    ПС.Т.И — инфраструктурная (платформенная, middleware)
    ПС.Т.ПТ — программно-технологическая (инструменты разработки, CI/CD)
    ПС.Т.П — прикладная (решает задачи предметной области)

14. Вид деятельности / предметная область:
    (примеры: Управление доступом, Биллинг, Аналитика, Нотификации)

15. Владелец подсистемы: @team-handle
```

### Шаг 5 — Определить нужные архитектурные представления

На основе ответов определи, какие из 11 типов представлений нужны:

| Тип | Вопрос | Когда нужен |
|-----|--------|-------------|
| Д-1: Внешнее взаимодействие деятельности | Кто внешние участники и интерфейсы? | Всегда (Standard+) |
| Д-2: Внутреннее взаимодействие деятельности | Есть ли внутренние процессы/функции? | Enterprise |
| Д-3: Внешние потоки данных деятельности | Какие данные входят/выходят? | Всегда (Standard+) |
| Д-4а/б: Процесс деятельности | Нужно ли описать конкретный процесс? | По потребности |
| Д-5: Внутренние потоки данных | Сложный обмен данными внутри? | Enterprise |
| Д-6: Функциональная карта | Нужен список всех функций? | Standard+ |
| П-1: Внешнее взаимодействие подсистемы | Как компоненты взаимодействуют снаружи? | Всегда (Standard+) |
| П-2: Внутреннее взаимодействие компонентов | Сложная внутренняя архитектура? | Standard+ |
| Т-1: Схема типов и связей узлов | Какая инфраструктура? | Всегда (Standard+) |
| О-1: Схема связи слоёв | Нужна сквозная трассировка? | Enterprise |

```
16. Какие представления нужны для этой подсистемы?
    - Standard/Extended: по умолчанию Д-1, Д-3, П-1, Т-1 — уточни, если нужно добавить или убрать
    - Enterprise: выбери из полного списка выше; минимум Д-1, Д-3, П-1, Т-1
```

### Шаг 6 — Сгенерировать и записать результат

На основе всех ответов:

1. **Заполни раздел «Архитектурные слои»** в `initiatives/$ARGUMENTS/design.md`:
   - _Только для Enterprise:_ Таблицу классификации подсистемы + таблицы элементов для каждого слоя (используй правильные типы из `canonical-model/model.md`)
   - _Для Standard/Extended:_ Добавь краткое текстовое описание ключевых компонентов каждого слоя без детальных таблиц элементов

2. **Обнови таблицу «Архитектурные представления»** — проставь статусы `заполнено` / `н/п`

3. **Сгенерируй Mermaid-заготовки** для каждого нужного представления:

   **Д-1 (внешнее взаимодействие):**
   ```mermaid
   graph LR
     actor["Актор (тип)"]
     subgraph system ["Подсистема"]
       iface["Интерфейс деятельности"]
     end
     actor -->|"операция"| iface
     iface -->|"обслуживание"| actor
   ```

   **Д-3 (внешние потоки данных):**
   ```mermaid
   graph LR
     source["Источник"]
     subgraph system ["Подсистема"]
       obj["Объект деятельности"]
     end
     sink["Получатель"]
     source -->|"данные (атрибуты)"| obj
     obj -->|"данные (атрибуты)"| sink
   ```

   **П-1 (внешнее взаимодействие подсистемы):**
   ```mermaid
   graph LR
     subgraph subsystem ["Подсистема — Слой приложений"]
       comp["Компонент приложения"]
       iface["Интерфейс приложения"]
       comp -->|"реализует"| iface
     end
     consumer["Потребитель"]
     iface -->|"обслуживание"| consumer
   ```

   **Т-1 (схема узлов):**
   ```mermaid
   graph TB
     subgraph node1 ["Узел: роль"]
       sw["Системное ПО"]
       artifact["Артефакт"]
     end
     subgraph node2 ["Узел 2: роль"]
     end
     node1 <-->|"протокол:порт"| node2
   ```

4. **Если профиль enterprise** — также создай `initiatives/$ARGUMENTS/subsystem-classification.yaml`:
   ```yaml
   initiative: "$ARGUMENTS"
   classification:
     system_scale: "{ ответ из вопроса 12 }"
     subsystem_type: "{ ответ из вопроса 13 }"
     subsystem_owner: "{ ответ из вопроса 15 }"
     activity_domain: "{ ответ из вопроса 14 }"
   architecture_views:
     - type: "Д-1"
       status: "заполнено"
       path: "design.md#д-1"
     - type: "Д-3"
       status: "заполнено"
       path: "design.md#д-3"
     - type: "П-1"
       status: "заполнено"
       path: "design.md#п-1"
     - type: "Т-1"
       status: "заполнено"
       path: "design.md#т-1"
   ontology_ref: "domains/is-ontology/canonical-model/model.md"
   ```

## Правила

- Используй **только типы элементов** из `canonical-model/model.md` (Таблица элементов)
- Используй **только типы взаимосвязей** из `relationship-catalog/relationships.md`
- Диаграммы — **только Mermaid** (graph LR / graph TB / C4Context)
- Не изобретай новые типы — если не подходит, уточни у архитектора
- После заполнения: `make validate` должен пройти без ошибок
- Не трогай другие разделы design.md (цели, контракты, качество)
