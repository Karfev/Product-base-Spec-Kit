# 00. Зачем SpecKit

> **Предыдущий файл:** — | **Следующий:** [01-glossary.md](./01-glossary.md)

## TL;DR

SpecKit — это **machine-readable контракт на изменения** в B2B SaaS-платформе.
Вместо того чтобы PRD жил в Notion, требования в Jira, OpenAPI в Confluence, ADR в личном репозитории
архитектора, а SLO — в Grafana, всё это лежит в **одной папке инициативы** и проверяется CI на каждый PR.

Если REQ-AUTH-001 в `requirements.yml` ссылается на `/api/auth/login` в OpenAPI и на тест
`auth.spec.ts::REQ-AUTH-001`, — связь обязана существовать, иначе CI падает.

## Что болит без SpecKit

Любой B2B SaaS, проживший 2-3 года, накапливает следующие проблемы:

1. **Расхождение между PRD и кодом.** PRD написан в Q1, код выкатили в Q3, в Notion никто не вернулся.
   Через 6 месяцев новый разработчик не понимает, **зачем** этот эндпойнт.
2. **Контракты живут отдельно от требований.** OpenAPI генерится из кода, требования в Jira, и связи
   между ними — устные. При breaking change узнаём от клиента в Slack.
3. **ADR-кладбище.** Архитектурные решения либо не пишутся, либо лежат в личных репозиториях.
   Через год команда наступает на те же грабли.
4. **SLO как декорация.** Метрики есть в Grafana, но нет привязки к требованиям —
   когда падает latency, непонятно, какое требование нарушено и кому эскалировать.
5. **Compliance-аудит = квест.** Для SOC2/ISO нужно показать «вот требование → вот тест → вот SLO».
   Без сквозной трассировки это занимает недели и собирается вручную.

## Что даёт SpecKit

Пять слоёв (L0–L5) с чёткими ролями и единым источником истины для каждого артефакта:

```text
L0  .specify/memory/constitution.md     Governance: правила игры, ID, CI gates
L1  domains/{domain}/                   Domain: glossary, canonical model, events
L2  products/{product}/                 Product: arch, NFR baseline, product ADR
L2.5 services/{service}/                Service: contracts, SLO, billing
L3  initiatives/{INIT}/                 Initiative: prd, requirements, contracts, ops
L4  .specify/specs/{NNN}/               Feature: spec → plan → tasks → implement
L5  evidence/                           Evidence: CI-генерируемые отчёты, RTM
```

**Главное конкретное преимущество для разработчика:**

- Открыл папку инициативы → за 5 минут понимаешь зачем (PRD), что (requirements.yml),
  как (design.md, contracts/), готова ли к продакшену (ops/prr-checklist.md).
- 32 slash-команды (`/speckit-*`) автоматизируют рутину: scaffold, валидация, генерация контрактов, интерактивный туториал.
- На каждом коммите CI проверяет, что REQ-IDs не оторвались от тестов и контрактов.

## Когда SpecKit не нужен

Будем честны — это не серебряная пуля.

| Сценарий | Подходит SpecKit? | Альтернатива |
|---|---|---|
| Стартап на 5 человек, MVP, ищем product-market fit | ❌ Overhead убьёт скорость | Notion + GitHub issues |
| Внутренний tooling без внешних потребителей | ⚠️ Только если несколько команд зависят | README + ADR в репо |
| Зрелый B2B SaaS, 20+ инженеров, enterprise-клиенты | ✅ Каноническое применение | — |
| Большая система с regulatory compliance (FinTech, MedTech) | ✅ Особенно Extended/Enterprise | — |
| Legacy-система без API-first подхода | ⚠️ Adoption будет долгим | Начать с нескольких новых инициатив |
